## 常见错误

| 错误 | 修正 |
|---|---|
| 把 `AGENTS.md` 写成长篇手册 | 保持为约束地图，细节放入 `docs/`。 |
| 有业务想法就直接写代码 | 先把业务想法转成约束和契约。 |
| 盲信旧文档 | 用代码、测试、manifest、配置反向验证。 |
| 新增 Tauri/Electron 命令只改一层 | 同步命令注册、权限/preload/wrapper、UI 调用点和测试。 |
| 只依赖源码字符串测试 | 用它守约束，再补行为测试覆盖逻辑和流程。 |
| 把平台签名当静态密钥保存 | 使用页面上下文或批准的 RPA 流程，并脱敏证据。 |
| 把真实凭据写进项目规则 | 替换为占位符，并标记需要轮换。 |
| 探索阶段直接跑真实写操作 | 先只读探针；真实写入前必须明确确认。 |
| 只列约束不列检查 | 给每条硬约束增加反馈机制。 |
| 相信 agent 自述完成 | 用命令输出、review 证据或人工确认验证。 |
| 长任务结束不留交接 | 更新 `task_plan.md` 或 `docs/progress.md`，写明下一步安全动作。 |
| 对话结束不写记忆文档 | 在 `docs/memory/` 新建带时间文件名的记录，写清本次对话内容和完成情况。 |
| 需求梳理时擅自加功能 | 把新增想法标为可选/后续，获得用户确认后才能进入范围。 |
| 多 agent 并发后直接宣布完成 | 先 review、合并、跑整体验收，再对照原始需求确认无越界功能。 |
| 一直修同一个问题但没有结果 | 第 3 次失败后停止修改，复盘流程并向用户提问重新校准。 |
| 用测试通过替代用户结果 | 测试只是证据之一，最终必须按用户要的结果验收。 |

## 最终检查

一个 Harness Engineering 项目只有满足以下条件，才适合进入实现：

- 业务需求明确。
- 约束已写出。
- `AGENTS.md` 指向约束、文档、命令和必用 skills。
- 详细文档放在 `docs/`。
- 测试和验证命令已在编码前规划。
- 安全和真实操作风险有明确门禁。
- 硬约束都有反馈机制。
- 长周期工作有交接文档。
- 本次对话已写入 `docs/memory/`，文件名包含日期和时间。
- 需求范围没有被擅自扩展。
- 多 agent 并发产物已经合并并通过整体验收。
- 每个需求都有最终结果验收标准，并已按该标准验收。
- 同一问题没有在 3 次失败后继续盲修；已复盘并向用户重新确认。

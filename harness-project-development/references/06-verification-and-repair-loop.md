# 06. 验证与修复循环

## 最终结果验收

每个需求的开发都必须以用户要的最终结果作为验收标准。过程正确、测试通过、代码完成、agent 自述完成，都不能替代最终结果验收。

最终验收需要回答：

- 用户原本要的业务结果是什么？
- 当前产物是否真实完成该结果？
- 验收使用了哪些命令、测试、浏览器/桌面端操作、真实样例或人工确认？
- 是否新增了用户没有要求的无关范围？
- 还有哪些风险或未验证项？

## 声称完成前必须检查

声明完成前必须：

- 重新读取业务需求和约束。
- 按用户要的最终结果验收，而不是只看代码、测试或中间产物。
- 对照原始需求确认没有擅自加入无关范围。
- 检查 `AGENTS.md` 是否仍与实现一致。
- 检查每条硬约束是否有反馈证据，或已列为开放风险。
- 运行 `AGENTS.md` 中列出的相关命令；如果不能运行，说明具体原因。
- 把 agent 自己说的“完成”视为未验证，直到有命令输出、测试、review 证据或人工确认支持。
- 根据发现更新 docs、plans、release notes 和 tests。
- 写入或更新 `docs/memory/YYYY-MM-DD-HHmm-<topic>.md`，记录本次对话内容、完成情况、验证证据、未完成事项和下一步。
- 把重复错误转成机械检查。

## 同一问题 3 次失败门禁

同一个问题连续修复 3 次仍未达到最终结果时，停止继续改代码。

处理规则：

- 第 1 次修复失败：回到 `systematic-debugging`，重新确认复现、日志、输入输出和假设。
- 第 2 次修复失败：对照开发文档、架构契约、测试设计和最终结果验收标准，检查是否修错层级或验收标准不完整。
- 第 3 次修复仍失败：停止继续修改，向用户提问，重新确认目标结果、失败现象、可接受方案和优先级。
- 未完成上述复盘和提问前，不得继续堆第 4 个修复。

复盘要检查这些环节：

- 需求是否理解错。
- 最终结果验收标准是否不完整。
- 架构契约是否缺失或冲突。
- 计划是否把任务拆错层。
- 实现是否只修了表面。
- 测试是否没有覆盖真实失败。
- 验证环境是否和用户环境不一致。
- 外部平台、权限、账号、数据或网络是否改变。

## 失败反馈循环

- 如果约束被误解，重写约束。
- 如果 bug 逃过测试，新增测试或验证步骤。
- 如果密钥、token、Cookie 或凭据出现在文档/日志中，轮换并新增扫描。
- 如果外部平台流程因签名、风控或 403 失败，更新 `docs/automation-flows.md` 和 RPA/API 判断规则。
- 如果未来 agent 迷路，改进 `AGENTS.md`、`task_plan.md` 或文档索引。

## 验证矩阵

```markdown
## 验证矩阵

| 验证项 | 命令/方法 | 覆盖约束 | 预期结果 | 实际结果 |
|---|---|---|---|---|
| 单元测试 | `npm test` | 业务逻辑 | 通过 |  |
| 构建 | `npm run build` | 构建可交付 | 通过 |  |
| Secret scan | `rg "<pattern>"` | 无真实凭据 | 无命中 |  |
| 手动验收 | [步骤] | 最终结果验收 | 达到用户结果 |  |
```

验证失败时，不要改验收标准来适配实现；先定位根因，再决定修复、改计划或向用户确认。

## 可运行的约束验证脚手架

约束反馈映射只是表格；要让它有牙齿，必须落成项目里能跑的脚本，否则约束只是 prose。初始化或新增硬约束时：

- 复制 `references/templates/verify.py` 到目标项目的 `scripts/verify.py`。
- 参照 `references/templates/constraints.example.json`，在目标项目建立 `docs/constraints.json`，把每条硬约束写成 `{id, constraint, check, evidence}`；`check` 是退出码 0 表示通过的命令。
- 把 `python scripts/verify.py` 写入 harness 地图的权威命令和验证矩阵，作为声明完成前必跑的门禁。
- 任何没有 `check` 命令的约束，verify 会标为 UNENFORCED 并使整体失败——这正是“约束没有反馈机制就不算落地”的机械兑现。

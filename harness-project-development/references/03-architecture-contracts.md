# 03. 架构契约

## 原则

每个边界都要在实现前定义契约。契约是让不同 agent、不同模块和不同阶段可以协作的接口说明；没有契约就并行开发，容易出现文件冲突、接口漂移和验收不一致。

## 必须定义的边界

按项目实际类型选择：

- Web：路由、状态归属、API client、表单校验、权限状态、面向用户的错误状态。
- Tauri：Rust command、权限配置、前端 wrapper、capability 配置、打包资源边界。
- Electron：main handler、preload API、renderer service、session/partition、沙箱配置。
- Cloud：路由、认证中间件、模块文件、环境变量、限流、日志、部署脚本。
- 数据库：schema、索引、RLS/资源级授权、迁移和回滚路径。
- Worker/RPA：输入 schema、浏览器状态、页面上下文规则、超时/重试策略、最终机器可读结果。
- AI/生图：模型选择、输入 schema、参考图处理、输出存储、失败重试、成本和内容边界。
- 运营工具：Excel/表格格式、字段映射、导入导出规则、人工确认点、结果复核方式。

## 契约模板

```markdown
## 契约：[边界名称]

### 责任
- 该模块负责：
- 该模块不负责：

### 输入
- 字段：
- 来源：
- 校验：

### 输出
- 字段：
- 写入位置：
- 用户可见结果：

### 错误与回滚
- 可恢复错误：
- 不可恢复错误：
- 用户提示：
- 回滚/重试：

### 安全
- 认证：
- 授权：
- 密钥/Token：
- 日志脱敏：

### 验证
- 单元测试：
- 契约测试：
- 集成测试：
- 手动验收：
```

## 源码约束测试

架构契约中“必须同步修改”的规则，应尽量转成机械检查：

- Tauri command 变更必须同步 `capabilities`、前端 wrapper 和调用点。
- Electron IPC 变更必须同步 main handler、preload 暴露、renderer service 和类型定义。
- API route 变更必须同步 client、schema、鉴权中间件和错误处理。
- 数据库 schema 变更必须同步迁移、类型、查询和授权策略。
- 新增环境变量必须同步 `.env.example`、部署文档和启动校验。

源码约束测试不替代行为测试。它用来阻止结构性漏改；业务正确性仍需要单元、集成、端到端或手动验证。

## 架构决策记录

先在 `docs/architecture.md` 落地具体技术栈：语言、运行时、前端框架、客户端壳、后端、数据库、关键库和包管理器的实际名称与版本，并把同一份当前栈（精简版）同步进 `AGENTS.md` 的技术栈栏。

当存在关键取舍时，写入 `docs/architecture.md` 或 ADR：

- 为什么选择 Web、Tauri、Electron、纯脚本或云服务。
- 哪些能力放在本地，哪些放在服务端。
- 哪些外部平台只能走 RPA，哪些可以走 API。
- 哪些数据可以缓存，缓存失效规则是什么。
- 哪些真实操作必须人工确认。

决策记录要短，重点是未来 agent 为什么不能随意改边界。

## 常见架构风险

| 风险 | 约束 |
|---|---|
| 只改 UI，不改命令/权限/API | 在计划中列出跨边界同步文件，并加源码约束测试。 |
| 把服务端密钥放进桌面端 | 服务端 key 只能在服务端或受控环境使用。 |
| 过早抽象多平台 | MVP 只实现已确认平台，扩展点写契约，不写未确认功能。 |
| RPA 与 API 混用无边界 | 每个平台流程必须写清 API 型、RPA 型、混合型或签名信封型。 |
| 错误状态只写 console | 用户可见流程必须有明确错误提示和恢复路径。 |

# 08. 记忆文档与会话记录

## 目标

让项目在多轮对话、多 agent、长周期开发中保留上下文。每次调用本 skill 完成一次项目对话或任务后，都要把本次对话内容和完成情况写入项目内的记忆文档，方便下一次接手时快速恢复上下文。

## 存放位置

默认使用：

```text
docs/memory/
```

如果目标项目已经有明确的记忆目录，可以沿用现有目录，但必须满足两个条件：

- 新文件名包含当前日期和时间。
- `AGENTS.md` 或 `docs/progress.md` 能索引到该目录。

## 文件命名

使用当前本地时间创建文件：

```text
docs/memory/YYYY-MM-DD-HHmm-<topic>.md
```

规则：

- 文件名里的日期和时间必须用系统命令实时获取，不能凭记忆或推测：
  - 类 Unix / Git Bash：`date +%Y-%m-%d-%H%M`。
  - PowerShell：`Get-Date -Format 'yyyy-MM-dd-HHmm'`。
- 示例：`docs/memory/2026-06-22-1530-project-init.md`。
- 同一分钟内需要多次创建时追加秒：`docs/memory/2026-06-22-153045-debug-loop.md`。
- `<topic>` 用简短英文或拼音 slug；未知主题用 `session`。
- 不要覆盖旧记忆文档；新对话、新阶段、新任务创建新文件。
- 同一次长会话内可以持续更新同一篇记忆文档，避免同一任务碎片化。

## 何时读取

开始项目级工作前读取：

- `AGENTS.md`
- `docs/progress.md` 或 `task_plan.md`
- `docs/memory/` 中最近 1-3 篇与当前任务相关的记忆文档

记忆文档只提供上下文，不替代当前代码、测试、配置和用户最新指令。如果记忆文档与可运行事实冲突，以当前代码和用户最新确认优先。

## 何时写入

在最终回复用户前写入或更新本次记忆文档。至少在这些场景必须写：

- 初始化项目或创建 Harness 文档后。
- 完成需求梳理、架构设计、实施计划或验收矩阵后。
- 修改代码、文档、配置、测试或脚本后。
- 多 agent 并发开发合并后。
- 调试失败、同一问题多次修复、向用户重新确认后。
- 任务未完成但需要交接时。

## 必写内容

```markdown
# YYYY-MM-DD HH:mm - <topic>

## 用户原始需求
- 

## 本次理解
- 

## 本次完成情况
### 已完成
- 

### 未完成
- 

### 阻塞/待确认
- 

## 修改文件
- 

## 验证证据
- 命令/方法：
- 结果：
- 无法验证项：

## 需求与约束变化
- 

## 下一步
- 

## 风险与注意事项
- 
```

## 记录边界

- 不记录真实密钥、Cookie、token、密码、连接串、平台签名或完整敏感日志。
- 不把未确认的想法写成已确认需求。
- 不复制大段代码或长日志；写路径、摘要和验证证据即可。
- 不用记忆文档替代 `requirements.md`、`architecture.md`、`verification.md` 或实施计划。
- 记忆文档要写清事实状态：完成、未完成、无法验证、等待用户确认，不能用模糊表述掩盖风险。

# 01. 需求与约束

## 目标

把用户的业务需求转成可执行、可验证、可沉淀的项目约束。Harness Engineering 的起点不是写代码，而是先明确 agent 允许做什么、禁止做什么、哪些证据可以证明结果正确。

## 读取当前 Harness

从目标项目根目录开始，优先读取：

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `docs/README.md`
- `docs/*索引*.md`、`docs/*总览*.md`、`docs/*context*.md`
- `task_plan.md`、`notes.md`
- 包管理配置、Tauri/Electron 配置、云服务入口、数据库 schema

然后用代码、测试、manifest、配置和实际命令反向验证文档。当前代码和测试通常比过期文档更可靠。

## 项目简报

设计前先写简短项目简报：

```markdown
## 业务需求
- 用户是谁：
- 要完成的工作：
- 核心业务流程：
- 支持的平台：
- MVP 范围：
- 明确不做：
- 真实操作风险：
- 成功标准：
- 最终结果验收标准：
```

如果某个字段未知且会阻塞安全设计，先问用户。如果不阻塞，写明假设并继续。

## 需求边界

需求梳理和开发文档阶段必须守住原始需求边界：

- 不得擅自添加与用户初始需求无关的功能、页面、平台、角色、自动化流程或运营能力。
- 可以提出风险、缺口和可选增强，但必须标记为“可选/后续”，不能写进 MVP 或默认开发范围。
- 如果发现需求必须扩展才能完成核心目标，先向用户说明原因并获得确认，再写入约束或开发文档。
- 开发文档必须能追溯到用户确认过的业务需求、约束或明确假设。
- 每个需求必须写清最终结果验收标准，不能只写实现步骤或技术指标。

## 约束来源追踪

约束必须从完整初始需求逐条推导，不能凭空增加“看起来合理”的内容。先把用户给出的需求、目标、非目标、场景、平台、角色、验收结果完整拆开，再生成与这些内容一一挂钩的约束；不能只抓几个关键词套用通用项目模板。

推荐用下面的追踪表把需求、约束和验收绑在一起：

```markdown
## 需求-约束追踪表

| 初始需求片段 | 推导出的约束 | 推导理由 | 验收证据 |
|---|---|---|---|
|  |  |  |  |
```

规则：

- 每条约束都要能指回初始需求、用户确认、现有代码事实或明确风险假设。
- “初始需求片段”必须来自用户真实表达或经过用户确认的补充，不能由 agent 自己补写成需求。
- 如果是从事实推导出来的约束，写明事实来源文件或配置。
- 如果是风险假设，必须标记为假设，不得包装成用户要求。
- 如果约束和初始需求没有直接关系，先删掉，再决定是否需要向用户确认。
- 若发现原始需求不足以支撑该约束，先补问用户，而不是自己补一个“更完整的范围”。

## 约束必须回答的问题

1. 这个项目必须为业务完成什么？
2. 这个项目绝对不能做什么？
3. 哪些文件、模块、命令、服务、凭据和平台是权威来源？
4. 哪些操作需要二次确认、沙箱隔离或真实环境验证？
5. 哪些测试或检查能防止未来 agent 跑偏？
6. 哪个反馈机制能证明每条硬约束被遵守？
7. 用户要的最终结果是什么，如何用最终结果验收？

## 项目约束结构

```markdown
## 项目约束

### 业务约束
- 目标用户、支持工作流、平台范围、MVP 边界。

### 架构约束
- 前端、桌面壳、后端、数据库、存储、队列、worker、部署边界。

### 集成约束
- 外部 API、RPA/页面自动化、浏览器上下文、签名、限流、重试、幂等、失败处理。

### 安全约束
- 认证、授权、密钥、Cookie、token、日志、本地存储、服务端 key、生产 HTTPS、最小权限。

### 测试约束
- 必需的单元测试、契约测试、源码约束测试、集成测试、浏览器测试、桌面端测试、云端测试、安全测试。

### 文档约束
- 哪些内容写入 AGENTS.md；哪些内容放入 docs/、memory/、references/、计划、发布说明。

### 运维约束
- 开发命令、构建命令、发布/更新流程、环境设置、回滚、监控、人工确认。
```

约束要具体到未来 agent 能判断某个修改是否被允许。

## 约束反馈映射

约束草案之后，必须创建反馈映射：

```markdown
## 约束反馈映射

| 约束 | 反馈机制 | 证据产物 |
|---|---|---|
| 文档和代码中不能出现真实密钥 | secret scan / 安全测试 | 命令输出 |
| Tauri command 变更必须同步权限和前端 wrapper | 源码约束测试 | 测试文件 + 通过结果 |
| 外部平台写操作必须二次确认 | UI/源码测试 + 人工确认门禁 | 测试输出 + 确认记录 |
| AGENTS.md 保持短地图 | 文档结构/长度检查 | AGENTS.md patch 摘要 |
```

如果某条硬约束没有反馈机制，就把反馈机制加入计划；如果暂时做不到，列入开放风险。

## 约束质量标准

- 约束能被 future agent 直接执行或判断。
- 每条硬约束都有反馈机制。
- 约束不包含真实凭据、Cookie、token、连接串或平台签名。
- 约束不把可选增强伪装成 MVP。
- 约束以最终业务结果为准，而不是以代码写完为准。

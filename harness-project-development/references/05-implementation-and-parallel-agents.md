# 05. 实现与多 Agent 并发

## 计划优先

先写计划，再写代码。每个计划任务必须包含：

- 要创建或修改的文件。
- 实现前要新增的测试。
- 预期 RED 阶段失败原因。
- 最小实现步骤。
- 验证命令。
- 最终结果验收方式。
- 真实系统的回滚或安全说明。

计划不能使用 `TBD`、`TODO`、`后续补充`、`适当处理` 这类占位表达。

## TDD 实现顺序

推荐顺序：

1. 针对文档、路由、命令、权限、环境边界的契约测试/源码约束测试。
2. 针对纯逻辑和 payload 构造的单元测试。
3. 针对跨进程行为的集成测试或 worker 测试。
4. 最小实现。
5. 目标验证。
6. 更大范围回归和构建检查。

实现功能或修复 bug 时，默认使用 `test-driven-development`：先写失败测试，确认失败原因正确，再写最小实现并验证通过。

## 不扩大开发范围

根据确定好的开发文档进行代码编写时，不得擅自新增与初始需求无关的能力。

- 未确认的新功能只能写入“可选/后续”，不能进入默认实现。
- 子 agent 不能自行改变产品范围。
- 发现文档缺口时，先补充约束或向用户确认，再实现。
- 每个任务都要能追溯到需求、约束、计划或用户确认。

## 多 Agent 并发开发

只有在开发文档和实施计划已经确认后，且任务确实属于大项目、大重构、大功能、复杂集成或难任务时，才使用 Superpowers 多 Agent 子代理并发开发。小任务、小功能、局部文档或少量代码调整，不启动多 agent，直接在当前会话中修改并做最小验证。

适用条件：

- 已有明确开发文档、架构契约、任务拆分和验收标准。
- 任务之间边界清晰，写入文件或模块尽量不重叠。
- 每个子任务都有自己的输入、输出、测试命令和完成标准。
- 用户没有要求当前会话单线程执行。
- 任务规模足够大，单 agent 执行会明显增加遗漏、冲突或上下文压力。

不适用条件：

- 单文件或少量文件的小改动。
- 需求已经很明确且无需跨模块设计。
- 只是文案、样式、配置说明或轻量 bugfix。
- 任务无法拆出清晰责任边界，强行并发会增加合并成本。

执行规则：

- 使用 `dispatching-parallel-agents` 判断哪些任务可以并行。
- 如果计划适合拆给子代理实现，使用 `subagent-driven-development` 或项目规定的多 agent 流程。
- 给每个子代理明确责任边界、允许修改文件、禁止修改文件、测试命令和交付格式。
- 子代理完成后，不能直接视为项目完成；必须先 review 子代理改动，再合并。
- 合并时检查接口契约、数据模型、权限、路由、UI 状态、测试和文档是否一致。
- 任何冲突、重复实现、遗漏测试或越界功能，都必须先修正再进入总体验收。

## 并发后的统一验收

- 跑每个子任务的局部验证。
- 跑整体验证矩阵中的测试、lint、构建、源码约束、安全检查或手动验收。
- 对照原始需求和开发文档逐项验收，确认没有擅自新增无关功能。
- 更新 `AGENTS.md`、`docs/`、`task_plan.md` 或 `docs/progress.md`，记录合并结果和剩余风险。
- 只有整体验收通过后，才允许声明完成。

## 子任务交付格式

```markdown
## 子任务结果
- 任务：
- 修改文件：
- 测试命令：
- 测试结果：
- 与契约的对应关系：
- 未完成/风险：
- 是否触及需求外范围：
```

主 agent 汇总时必须独立检查 diff、测试和约束，不直接相信子 agent 自述。

# 04. 外部平台、RPA 与安全

## 外部平台与 RPA 规则

涉及平台自动化时，先确认操作类型和风险，不要在探索阶段直接执行真实写入。

执行规则：

- 先用只读脚本、测试账号、沙箱资源或已捕获证据做探针。
- 将流程分类为 API 型、页面/RPA 型、混合型，或平台专属信封/签名型。
- 平台需要浏览器生成签名时，优先使用页面上下文请求。
- 写接口遇到风控、403、签名失败或动态风控时，改用 UI/RPA。
- 日志中必须脱敏 Cookie、token、签名、用户 ID、手机号、地址、订单号等敏感信息。
- 真实写入、破坏性动作、批量操作、合同发送、订单变更、不可逆平台变更前，必须人工确认。
- 把脱敏后的发现保存到 `docs/automation-flows.md` 或平台专项文档。

## 安全红线

绝不把真实密钥写入 `AGENTS.md`、README、docs、源码、测试、日志、截图、示例或 skill。

只使用占位符：

- `MONGODB_URI=...`
- `SUPABASE_SERVICE_ROLE_KEY=...`
- `API_TOKEN=...`
- `OSS_ACCESS_KEY_SECRET=...`
- `SERPER_API_KEY=...`

如果已有文档含真实凭据，不要复述；标记需要轮换，并替换为占位符。

## 安全默认值

- 桌面安装包不能内置 service-role key 或供应商密钥。
- 生产云端 API 必须使用 HTTPS，认证配置缺失时 fail closed。
- Cookie 和 token 要最小化；本地存储时加密；日志脱敏；登出时清理。
- 授权要做到资源级，不只是“已登录”。
- 会写数据的 smoke test 默认使用隔离测试资源。
- API key 优先放入用户级环境变量、密钥管理器或平台 secret，不放进仓库。

## 何时调用 security-auditor

以下情况必须调用 `security-auditor`：

- 登录、认证、授权、会话、Cookie、token。
- 支付、合同、订单、商家数据、客户数据。
- 外部平台自动化、RPA、抓包接口、签名。
- 云端部署、环境变量、密钥、日志、对象存储。
- 发布前安全检查。

## 安全反馈机制

建议至少包含：

- secret scan：检查真实 key、token、连接串。
- 依赖检查：检查高危依赖或已知漏洞。
- 权限检查：桌面端 capability、Electron sandbox、API 中间件。
- 日志检查：错误日志和调试日志不泄露凭据。
- 真实操作门禁：写操作必须有人确认或使用隔离资源。

安全约束没有反馈机制时，不允许视为已经落地。

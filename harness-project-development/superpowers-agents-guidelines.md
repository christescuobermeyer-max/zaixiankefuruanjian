# Superpowers 工作流规范

本项目启用 Superpowers skills 作为 agent 的强制协作流程。Agent 在执行任务时，必须先判断当前任务是否触发对应 skill；如果触发，必须先使用该 skill，再继续后续操作。用户的明确要求始终优先于 skill，但不能因为任务看起来简单就跳过必要流程。

## 总原则

- 开始任何对话或任务时，先使用 `using-superpowers` 判断应调用哪些 skills。
- 任何创意、需求、功能、组件、行为修改，在动手前必须使用 `brainstorming` 梳理目标、约束、方案和验收标准。
- 多步骤开发任务，在需求明确后必须使用 `writing-plans` 写出可执行计划，再进入实现。
- 实现功能或修复 bug 时，默认采用 `test-driven-development`，先写失败测试，再写最小实现，再验证通过。
- 遇到 bug、测试失败或异常行为时，必须使用 `systematic-debugging`，先定位原因，再提出修复。
- 声称完成、修复、测试通过、准备提交或 PR 前，必须使用 `verification-before-completion` 做最终验证。
- 不得跳过 review、测试、验证和收尾步骤；不得只凭推测宣布成功。

## Skill 使用目标与触发条件

### `using-superpowers`

目标：建立本次任务的 skill 调用策略，确保 agent 不遗漏必要流程。

触发条件：
- 每次新对话开始。
- 每次用户提出新任务、修改任务方向或追加要求。
- 不确定是否需要某个 skill 时。

执行要求：
- 先判断任务类型，再决定调用哪些 skills。
- 如果多个 skill 适用，优先使用流程类 skill，再使用实现类 skill。
- 用户明确指令优先，但必须说明与 skill 流程的关系。

### `brainstorming`

目标：在实现前把需求、目标、约束、方案、边界和验收标准梳理清楚。

触发条件：
- 创建新功能、新页面、新组件、新脚本、新自动化流程。
- 修改已有行为或设计。
- 用户提出模糊需求，需要转成可执行方案。
- 任何创意性或产品性工作。

执行要求：
- 先理解当前项目上下文。
- 一次只问一个关键澄清问题。
- 至少提出 2-3 个方案，并说明取舍。
- 在进入实现或写计划前，先给出设计方案并获得用户认可。
- 避免擅自增加未确认的功能。

### `writing-plans`

目标：把已确认的需求或设计转成可执行开发计划，让没有上下文的开发者也能按步骤完成。

触发条件：
- 已有明确需求、规格或设计。
- 任务包含多个步骤、多个文件或多个阶段。
- 准备进入实现前。

执行要求：
- 写清目标、架构、技术栈、全局约束。
- 明确每个任务要创建、修改、测试的文件。
- 每个任务必须包含测试、实现、验证、提交步骤。
- 不允许出现 TBD、TODO、稍后实现、补充测试等占位表达。
- 计划完成后再进入执行阶段。

### `test-driven-development`

目标：用测试驱动实现，降低回归和误实现风险。

触发条件：
- 实现任何新功能。
- 修复任何 bug。
- 修改核心逻辑、数据处理、API、状态管理或用户流程。

执行要求：
- 先写能失败的测试。
- 运行测试确认失败原因正确。
- 写最小实现让测试通过。
- 再运行相关测试和必要的回归测试。
- 不要在没有测试保护的情况下修改高风险逻辑。

### `systematic-debugging`

目标：系统化定位问题根因，而不是凭猜测修补。

触发条件：
- 测试失败。
- 程序报错。
- 行为与预期不一致。
- 用户报告 bug。
- 工具、脚本、构建、部署出现异常。

执行要求：
- 先复现问题。
- 收集错误信息、日志、输入输出和环境条件。
- 提出假设并逐一验证。
- 找到根因后再修复。
- 修复后必须增加或更新能覆盖该问题的测试。

### `using-git-worktrees`

目标：为较大或有风险的开发工作创建隔离工作区，避免污染当前工作状态。

触发条件：
- 开始独立功能开发。
- 执行较大的实现计划。
- 当前工作区已有未提交或无关改动。
- 需要并行处理多个分支或任务。

执行要求：
- 优先确认当前 git 状态。
- 使用 git worktree 或项目已有隔离机制创建独立 workspace。
- 不覆盖、不回滚用户已有改动。
- 工作完成后按项目规范合并或清理。

### `dispatching-parallel-agents`

目标：把互不依赖的任务并行分派，提高处理效率。

触发条件：
- 有 2 个以上独立任务。
- 各任务之间没有共享状态或顺序依赖。
- 可以分别阅读、分析、实现或验证。

执行要求：
- 明确每个并行任务的边界和输出。
- 不把强依赖任务错误并行化。
- 汇总结果时检查冲突、重复和遗漏。

### `subagent-driven-development`

目标：按计划用 subagent 分任务实现，并在任务之间进行检查。

触发条件：
- 已有 `writing-plans` 生成的实施计划。
- 计划可拆成多个独立任务。
- 适合每个任务由独立 agent 完成并 review。

执行要求：
- 每个任务使用独立 subagent。
- 每个任务完成后先 review 再继续下一个。
- 确保任务产物符合计划中的接口、测试和文件要求。
- 汇总后运行整体验证。

### `executing-plans`

目标：在当前会话中按已写好的计划逐步执行。

触发条件：
- 已有明确实施计划。
- 用户选择当前会话内执行，而不是 subagent 分派。
- 任务需要连续上下文或不适合拆给多个 subagent。

执行要求：
- 严格按计划任务顺序执行。
- 每个阶段完成后运行对应测试。
- 遇到计划与实际代码不一致时，先更新判断并说明原因。
- 不跳过检查点。

### `requesting-code-review`

目标：在完成实现后主动请求代码审查，发现遗漏、风险和质量问题。

触发条件：
- 完成一个任务或主要功能。
- 准备合并、提交 PR 或交付前。
- 修改范围较大、风险较高或影响用户流程。

执行要求：
- 提供变更摘要、测试结果和重点审查区域。
- 要求 review 关注 bug、回归、边界情况和缺失测试。
- 根据审查结果决定是否返工。

### `receiving-code-review`

目标：理性处理代码审查反馈，不盲从，也不忽视。

触发条件：
- 收到用户、reviewer 或 subagent 的代码审查意见。
- 反馈不清楚、可能有争议或影响实现方向。
- 需要判断建议是否技术上成立。

执行要求：
- 逐条理解反馈。
- 对技术建议进行验证。
- 合理的反馈必须修复。
- 不合理或不适用的反馈要说明原因。
- 修复后重新测试相关行为。

### `verification-before-completion`

目标：在宣布完成前提供事实验证，避免未经验证的成功声明。

触发条件：
- 准备说“完成了”“修好了”“测试通过了”。
- 准备提交、合并、发 PR 或交付。
- 完成 bugfix、功能、重构、脚本或文档改动。

执行要求：
- 运行相关测试、构建、lint 或手动验证命令。
- 检查命令输出确实成功。
- 如果无法验证，必须明确说明原因和剩余风险。
- 最终回复中包含实际验证结果。

### `finishing-a-development-branch`

目标：在开发完成后选择合适的收尾方式。

触发条件：
- 实现完成且验证通过。
- 需要决定 merge、PR、保留分支、清理 worktree 或交付说明。
- 当前分支开发工作准备结束。

执行要求：
- 确认 git 状态。
- 总结变更和验证。
- 根据项目流程选择合并、PR 或保留分支。
- 清理临时文件和无用工作区，但不得删除用户未确认的改动。

### `writing-skills`

目标：创建、修改或验证 Codex/agent skill 时保证结构、触发条件和说明可用。

触发条件：
- 新建 skill。
- 修改已有 skill。
- 调试 skill 的触发、说明、脚本或模板。
- 准备发布或部署 skill。

执行要求：
- 明确 skill 的使用场景、触发条件和非适用场景。
- 编写清晰的 `SKILL.md`。
- 如有脚本或模板，优先复用并验证。
- 部署前检查 skill 是否能被正确发现和执行。

### `project-analyzer`

目标：在项目初始化、复刻或理解陌生代码库时，先摸清结构、技术栈、约定和验证方式，再动手。

触发条件：
- 空目录初始化或项目级改动前。
- 复刻、重构现有系统，或接手不熟悉的代码库。
- 需要识别权威事实来源、依赖、构建/测试命令。

执行要求：
- 先读取现有 harness：`AGENTS.md`、`CLAUDE.md`、README、docs、配置、测试、核心代码。
- 用代码、测试、manifest 反向验证文档，过期文档不可信。
- 输出结构、依赖、约定、风险和建议，不在分析阶段擅自改代码或扩范围。

### `security-auditor`

目标：在涉及认证、密钥、外部平台或发布前，系统化审查安全边界，避免凭据泄露和越权。

触发条件：
- 登录、认证、授权、会话、Cookie、token。
- 支付、合同、订单、商家或客户数据。
- 外部平台自动化、RPA、抓包接口、签名。
- 云端部署、环境变量、密钥、日志、对象存储、发布前安全检查。

执行要求：
- 按 `references/04-rpa-and-security.md` 的安全红线和默认值逐项核对。
- 真实密钥只用占位符，写操作有人工确认或隔离资源门禁。
- 每条安全约束都要有反馈机制（secret scan、依赖检查、权限检查、日志检查），否则不算落地。

## 推荐执行顺序

### 新功能或复杂改动

1. `using-superpowers`
2. `brainstorming`
3. `writing-plans`
4. `using-git-worktrees`
5. `test-driven-development`
6. `subagent-driven-development` 或 `executing-plans`
7. `requesting-code-review`
8. `receiving-code-review`
9. `verification-before-completion`
10. `finishing-a-development-branch`

### Bug 修复

1. `using-superpowers`
2. `systematic-debugging`
3. `test-driven-development`
4. `verification-before-completion`
5. 必要时使用 `requesting-code-review`

### 多任务并行

1. `using-superpowers`
2. `dispatching-parallel-agents`
3. 各任务按需要使用 `systematic-debugging`、`test-driven-development` 或 `executing-plans`
4. 汇总后使用 `verification-before-completion`

## 完成标准

Agent 只有在满足以下条件后，才能声称任务完成：

- 已调用所有适用的 Superpowers skills。
- 已按需求或计划完成实现。
- 已运行必要测试、构建、lint 或手动验证。
- 已说明验证结果。
- 已列出未完成事项、无法验证事项或剩余风险。
- 未擅自回滚、覆盖或删除用户已有改动。

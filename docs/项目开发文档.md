## 0. 安全声明（务必先读）

初始需求文档中明文包含以下敏感凭据，**严禁提交到任何代码仓库或前端可访问位置**：

- Supabase Access Token
- 云雾（yunwu.ai）LLM API Key

处置要求：

3. **LLM Key、MongoDB 连接串、外卖客服 system prompt 只存在于后端**，客户端永不接触。
4. 既然 Access Key / Token 已在明文文档中泄露，上线前**强烈建议轮换**全部凭据。
5. 本文档中所有凭据均以 `${占位符}` 形式出现。

## 1. 项目概述

桌面端 AI 在线客服助手。常驻 Windows 桌面顶部，默认隐藏；鼠标移动到屏幕顶部热区时下滑弹出聊天面板（Auto-hide Panel / Screen Edge Reveal）。用户登录后与后端托管的"外卖在线客服"大模型对话，所有对话内容保存到云端 MongoDB，并支持基于阿里云 OSS 的自动更新推送。

### 1.1 核心目标

| # | 目标 | 说明 |
|---|------|------|
| G1 | 桌面顶部自动隐藏面板 | 热区触发下滑显示，失焦自动上滑隐藏 |
| G2 | 接入第三方 LLM | 后端转发云雾 OpenAI 兼容接口，前端用 Vercel AI SDK 流式渲染 |
| G3 | 登录鉴权 | Supabase Auth 邮箱+密码登录，JWT 鉴权后端 API |
| G4 | 对话持久化 | 每轮对话原文保存到 MongoDB `zaixiankefu` 库 |
| G5 | 上下文自动压缩 | 累计 token 超阈值时自动摘要压缩历史 |
| G6 | 自动更新 | Tauri Updater ← 阿里云 OSS 独立子目录 |
| G7 | 多对话管理 | 新建对话时弹窗输入名称（建议填店铺名）区分上下文 |

### 1.2 非目标（YAGNI）

- 不做多租户/团队协作（单账号列表登录即可）。
- 不做移动端。
- 不做语音/图片多模态（首版纯文本）。
- 不做本地离线模型。

## 2. 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 桌面外壳 | **Tauri 2.x**（Rust） | 体积小、安全、原生窗口控制，支持 always-on-top / 无边框 / 全局事件 |
| 前端 | **React + TypeScript + Vite** | Tauri 默认 web 前端 |
| 聊天框架 | **Vercel AI SDK** + **prompt-kit** | 流式聊天 UI、消息渲染 |
| 鉴权 | **Supabase Auth (supabase-js)** | 邮箱密码登录，签发 JWT |
| 自动更新 | **@tauri-apps/plugin-updater** | 检查 OSS 上的 `latest.json` 并增量更新 |
| 后端 | **Node.js + TypeScript + Hono** | 轻量、原生流式（SSE）友好，与 AI SDK 同语言 |
| LLM 接入 | 云雾 `https://yunwu.ai/v1/chat/completions` | OpenAI 兼容，`gemini-3.1-flash-lite` |
| 数据库 | **MongoDB**（Sealos 云） | 库名 `zaixiankefu` |
| 部署 | Sealos Devbox（云服务器） | SSH 部署，PM2/容器常驻 |
| 对象存储 | 阿里云 OSS bucket `${ALI_OSS_BUCKET}` | 更新包独立子目录 `zaixiankefu/` |

> **后端选型理由**：需求前端已定为 Vercel AI SDK（TS 生态）。后端同样用 TS 可直接复用 AI SDK 的 `streamText` / OpenAI 兼容 provider，前后端类型共享、流式 SSE 对接零摩擦，团队心智负担最低。选 Hono 而非 Express 是因其原生支持 Web 标准 `Response`/流，部署形态灵活（Node / 边缘均可）。

## 4. 桌面面板交互（Auto-hide / Edge Reveal）

### 4.1 状态机

```
        ┌────────────┐   鼠标进入顶部热区 / 悬停图标   ┌────────────┐
        │  HIDDEN    │ ──────────────────────────────▶ │  REVEALED  │
        │ (面板上滑  │                                  │ (面板下滑  │
        │  屏幕外，  │ ◀────────────────────────────── │  完整显示) │
        │  顶部留图标)│   失焦 / 鼠标离开 + 延时 400ms   └────────────┘
        └────────────┘            ▲           │
                                  │  Pin 固定 │ 用户点击 📌
                                  └───────────┘ (PINNED：禁用自动隐藏)
```

### 4.2 实现要点（Tauri）

- 窗口：`decorations: false`、`alwaysOnTop: true`、`skipTaskbar: true`，宽度 720px（见 UI 文档），吸附屏幕顶部居中。
- 热区检测：Rust 侧监听全局鼠标位置（`tauri-plugin-positioner` + 定时轮询或系统钩子），当 `y <= 2px` 且 `x` 在面板水平范围内 → emit `reveal` 事件。
- 显示/隐藏动画：前端用 CSS `transform: translateY()` 过渡（240ms ease-out），由 Rust 事件驱动窗口 `show()/hide()` + 位置动画。
- 隐藏态保留一个 48×48 顶部居中悬浮图标窗口（或主窗口仅露出图标高度）；鼠标悬停 200ms 触发展开。
- Pin 模式：用户点击图钉后禁用自动隐藏，便于长时间客服工作。
- 失焦隐藏：监听 `WindowEvent::Focused(false)`，延时 400ms 后隐藏（避免误触）。

## 5. 后端 API 设计

所有接口（除健康检查）需 `Authorization: Bearer <Supabase JWT>`，由 `authMiddleware` 校验。

### 5.1 接口清单

| 方法 | 路径 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| GET | `/health` | 健康检查 | - | `{status:"ok"}` |
| GET | `/api/conversations` | 当前用户对话列表 | - | `Conversation[]` |
| POST | `/api/conversations` | 新建对话 | `{name}` | `Conversation` |
| GET | `/api/conversations/:id/messages` | 对话历史 | - | `Message[]` |
| POST | `/api/chat` | 流式对话（SSE） | `{conversationId, message}` | `text/event-stream` |
| DELETE | `/api/conversations/:id` | 删除对话 | - | `{ok:true}` |

### 5.2 鉴权中间件

```ts
// 校验 Supabase JWT：用 SUPABASE_JWT_SECRET 验签，取 sub 作为 userId
// 校验失败 → 401。userId 注入 ctx，用于隔离各用户对话数据。
```

### 5.3 `/api/chat` 处理流程

```
1. 校验 JWT → userId
2. 读取 conversation（校验归属 userId）
3. 取该对话历史消息 + 已有 summary
4. 估算累计 token；若超阈值 CONTEXT_TOKEN_LIMIT(默认 8000)：
     → 调 LLM 摘要较早消息 → 写回 conversation.summary
     → 仅保留最近 KEEP_RECENT_ROUNDS(默认 6 轮) 原文
5. 组装 messages = [system prompt] + [summary as system] + [recent] + [user new]
6. 调云雾 LLM（stream）→ SSE 流式回传前端
7. 流结束：把 user 原文 + assistant 完整回复写入 messages 集合，更新 tokenCount
```

### 5.4 外卖客服 System Prompt（后端持有，示例骨架）

```
你是「{店铺名}」的外卖在线客服助手。语气亲切、专业、简洁。
职责：处理订单咨询、催单、退款/缺漏餐、配送时效、菜品推荐、优惠活动说明。
规则：1) 不承诺无法兑现的赔付；2) 涉及退款金额先安抚并引导走平台流程；
3) 不泄露内部成本/系统信息；4) 无法解决时引导联系人工/平台客服。
输出：中文，必要时分点，避免冗长。
```

> 实际 prompt 文案在后端 `src/prompts/customer-service.ts` 维护，可按店铺动态注入名称。

## 6. 数据模型（MongoDB `zaixiankefu`）

### 6.1 `conversations`

```jsonc
{
  "_id": "ObjectId",
  "userId": "string",        // Supabase sub
  "name": "string",          // 对话名称（建议店铺名）
  "summary": "string",       // 压缩后的历史摘要，初始为空
  "tokenCount": 0,           // 累计 token 估算
  "createdAt": "ISODate",
  "updatedAt": "ISODate"
}
```
索引：`{userId:1, updatedAt:-1}`

### 6.2 `messages`

```jsonc
{
  "_id": "ObjectId",
  "conversationId": "ObjectId",
  "userId": "string",
  "role": "user | assistant",
  "content": "string",
  "tokens": 0,
  "createdAt": "ISODate"
}
```
索引：`{conversationId:1, createdAt:1}`

## 7. 上下文压缩策略

- 阈值：`CONTEXT_TOKEN_LIMIT = 8000`（可配）。token 用近似算法（中文按字符/1.6 估算，或引入 `gpt-tokenizer`）。
- 触发：发送前累计 token 超阈值。
- 压缩：把"最近 N 轮之前"的消息送 LLM 做摘要（"用要点概括以下客服对话，保留关键订单/诉求信息"），结果写入 `conversation.summary`，被摘要的原文仍保留在 `messages`（仅不再进上下文，便于追溯）。
- 重建上下文：`system prompt` + `（如有）summary 作为 system 背景` + `最近 N 轮原文` + `本次用户输入`。

## 8. 自动更新（Tauri Updater + 阿里云 OSS）

### 8.1 OSS 目录隔离（重要）

`shiping-1-2` bucket 已有其他项目更新文件，本项目**必须独立子目录**：

```
${ALI_OSS_BUCKET}/
└── zaixiankefu/                 ← 本项目专属，勿与他项目混淆
    ├── latest.json
    ├── zaixiankefu_1.0.0_x64-setup.nsis.zip
    └── zaixiankefu_1.0.0_x64-setup.nsis.zip.sig
```

### 8.2 `latest.json` 示例

```json
{
  "version": "1.0.1",
  "notes": "修复若干问题",
  "pub_date": "2026-06-28T00:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "<minisign 签名>",
      "url": "https://${ALI_OSS_BUCKET}.oss-accelerate.aliyuncs.com/zaixiankefu/zaixiankefu_1.0.1_x64-setup.nsis.zip"
    }
  }
}
```

### 8.3 流程

1. `tauri.conf.json` 配置 `plugin.updater.endpoints = ["https://${BUCKET}.oss-accelerate.aliyuncs.com/zaixiankefu/latest.json"]`，并配置 `pubkey`（minisign 公钥）。
2. 构建时用私钥签名产物，CI/脚本用 OSS SDK 上传产物 + `latest.json` 到 `zaixiankefu/` 子目录。
3. 客户端启动后定时检查；有新版本则下载校验签名 → 提示用户 → 重启更新。

> OSS 上传使用 `ali-oss` SDK，凭据从后端/CI 环境变量读取，绝不写入客户端。

## 12. 风险与对策

| 风险 | 对策 |
|------|------|
| 凭据已明文泄露 | 上线前轮换全部 Key/Token；迁移至环境变量 |
| 顶部热区误触发 | 加入 200ms 悬停延时 + Pin 固定模式 |
| LLM 流式中断 | SSE 断线重连 + 已存 user 消息可重试 |
| token 估算不准导致超长 | 引入真实 tokenizer，保留安全余量 |
| OSS 与他项目文件混淆 | 强制 `zaixiankefu/` 子目录前缀，发布脚本写死 prefix |
| 多用户数据串库 | 所有查询按 `userId` 过滤，后端强制隔离 |
